scripts/run_full_stage2_ddp.sh

Purpose

This is an isolated full-data Stage-2 launcher for PyTorch DistributedDataParallel (DDP). It deliberately reuses read-only Stage-1 weights and prebuilt graphs while writing to a separate workspace, allowing multi-GPU experiments without touching the main training run.

Check your understanding

  1. Why does this launcher reuse prebuilt graphs and a frozen Stage-1 checkpoint?

    Show answer

    It isolates the multi-GPU Stage-2 experiment from expensive preprocessing and from changes to the main Stage-1 or full-training workspace.

  2. What makes the launcher's output isolated from its read-only inputs?

    Show answer

    Inputs default to existing home-mounted data, graphs, and a snapshot, while checkpoints and W&B state are written below a separate S2_WORK_DIR.

  3. Why is DDP appropriate specifically for this full-data stage?

    Show answer

    The large Stage-2 workload can distribute disjoint samples across GPUs while synchronizing model gradients, reducing wall time compared with one process.

How it fits in

condor/full_stage2_mgpu.sub and condor/smoke_2gpu.sub stage this script beside train_ddp.py and seminar_csv.py. Inputs default to paths under the mounted home but can be overridden with FULL_DATA_DIR, FULL_GRAPHS_DIR, S2_WORK_DIR, STAGE1_CKPT, S2_BATCH, S2_ACCUM, and S2_EPOCHS. Output is $S2_WORK_DIR/ckpt_s2_ddp/ plus W&B logs.

Check your understanding

  1. How do full_stage2_mgpu.sub and smoke_2gpu.sub alter the same launcher for different experiments?

    Show answer

    They choose resources and environment overrides such as GPU count and epoch limit while staging the same launcher and DDP trainer.

  2. Why are the environment variables useful even though the script supplies defaults?

    Show answer

    They relocate large home-mounted inputs and outputs or adjust batch, accumulation, checkpoint, and epoch settings without editing the executable.

  3. Which artifact is expected after a successful run?

    Show answer

    The best-validation model is expected at $S2_WORK_DIR/ckpt_s2_ddp/karmadock_team002.pkl, with resume and logging state in the same isolated workspace.

Walkthrough

#!/usr/bin/env bash
# ...
set -euo pipefail
set -x

KD=/app/KarmaDock
export PYTHONPATH=/app/KarmaDock:${PYTHONPATH:-}
SUBMIT="$PWD"

Strict Bash behavior and trace logging are valuable for multi-process failures. The upstream installation is made importable; transferred files are expected at the job sandbox root.

DATA="${FULL_DATA_DIR:-$HOME/repro_test/full_data}"
GRAPHS="${FULL_GRAPHS_DIR:-$HOME/repro_test/work_full}"
S2WORK="${S2_WORK_DIR:-$HOME/stage2_mgpu_test/work}"
STAGE1_CKPT="${STAGE1_CKPT:-$HOME/stage2_mgpu_test/stage1_snapshot.pkl}"
BS="${S2_BATCH:-8}"; ACC="${S2_ACCUM:-2}"
EP="${S2_EPOCHS:-1000}"

mkdir -p "$S2WORK"
[ -f "$STAGE1_CKPT" ] || { echo "ERROR: Stage-1 checkpoint not found: $STAGE1_CKPT"; exit 3; }

Defaults keep large inputs in the home mount and all outputs in the isolated stage2_mgpu_test tree. The grouped { ...; } command produces a clear error and exit code 3 if the frozen initialization checkpoint is missing.

NGPU=$(nvidia-smi -L 2>/dev/null | wc -l); [ "${NGPU:-0}" -lt 1 ] && NGPU=1
RUN="full_stage2_ddp_$(date +%Y%m%d-%H%M%S)"
echo "=== Stage-2 DDP on $NGPU GPU(s); effective batch = $((NGPU*BS*ACC)); run=$RUN ==="

nvidia-smi -L emits one line per visible GPU. The arithmetic expression reports global effective batch as processes × per-process physical batch × accumulation steps. The timestamp makes the human-facing W&B base name unique. The fallback to one does not create a GPU; it selects the single-process branch and lets train_ddp.py decide CUDA versus CPU.

ARGS=( "$SUBMIT/train_ddp.py"
  --csv "$DATA/full_train.csv"   --graph_dir "$GRAPHS/full_train/graphs" --complex_dir "$GRAPHS/full_train/complex"
  --val_csv "$DATA/full_val.csv" --val_graph_dir "$GRAPHS/full_val/graphs"
  --out_dir "$S2WORK/ckpt_s2_ddp" --init_model "$STAGE1_CKPT" --pos_r 1
  --lr 1e-4 --weight_decay 1e-4 --patience 20 --epochs "$EP"
  --batch_size "$BS" --accum_steps "$ACC" --num_workers 3 --random_seed 42 --resume
  --wandb --wandb_project karmadock-seminar --wandb_run_name "$RUN" )

The array safely preserves argument boundaries. pos_r=1 activates combined docking/RMSD and MDN scoring training. This original version uses weight decay 1e-4 and patience 20. Explicit train/validation inputs preserve the curated split.

if [ "$NGPU" -gt 1 ]; then
  torchrun --standalone --nnodes=1 --nproc_per_node="$NGPU" "${ARGS[@]}"
else
  python3 -u "${ARGS[@]}"
fi
echo "=== Stage-2 DDP done: $S2WORK/ckpt_s2_ddp/karmadock_team002.pkl ==="

torchrun starts one Python process per GPU and supplies RANK, LOCAL_RANK, and WORLD_SIZE; train_ddp.py uses those to initialize NCCL and DDP. --standalone --nnodes=1 sets up rendezvous locally on one machine. A one-GPU allocation avoids process-group setup and uses the script's single-process path.

Check your understanding

  1. How is the reported global effective batch computed?

    Show answer

    It multiplies visible GPU processes by per-process physical batch S2_BATCH and accumulation windows S2_ACCUM.

  2. Why do --pos_r 1 and explicit validation paths matter for Stage 2?

    Show answer

    pos_r=1 enables combined coordinate and scoring learning, while the separate full-validation graphs preserve the curated train/validation split.

  3. What changes when only one GPU is visible?

    Show answer

    The launcher calls ordinary python3 instead of torchrun, avoiding process-group setup while retaining the same trainer arguments.

Gotchas / notes

  • nvidia-smi is used inside a pipefail script. If the command is absent or fails rather than merely listing zero devices, the assignment pipeline may terminate the script before the fallback runs.
  • --resume reuses $S2WORK/ckpt_s2_ddp/last.pt; reuse of the same output directory can therefore continue an earlier run even though RUN has a new timestamp.
  • Defaults BS=8, ACC=2 yield global batch 64 only with four GPUs; submit-file overrides are needed for other world sizes.

Check your understanding

  1. Why might the GPU-count fallback never run if nvidia-smi itself fails?

    Show answer

    The count is produced by a pipeline inside a pipefail script, so failure of nvidia-smi can terminate execution before the later zero-count fallback.

  2. How can a new W&B run name still resume old optimizer state?

    Show answer

    The timestamp changes the displayed name, but --resume reads last.pt from the reused checkpoint directory independently of that name.

  3. Why do the default batch settings reproduce effective batch 64 only on four GPUs?

    Show answer

    The defaults are batch 8 and accumulation 2, so the product reaches 64 only when world_size is four; other allocations need overrides.
