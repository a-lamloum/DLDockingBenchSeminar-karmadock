File-by-file guide

Repository root

File What it is
README.md The submission's main write-up: what the team built and why, the three pipelines, results tables, how to evaluate/reproduce, the paper training parameters, repository layout, and known issues.
Dockerfile Builds the reproducible image ahlamloum/karmadock-seminar:v6. Starts from miniconda, clones upstream KarmaDock, and unpacks the authors' PRE-PACKED conda environment (a conda-pack tarball from Zenodo) instead of solving karmadock_env.yaml — this sidesteps conda's dependency solver. It then pip-installs a few extras (prefetch_generator, rmsd, wandb), copies the team's scripts/, sets PYTHONPATH=/app/KarmaDock, and runs a build-time import check so a broken environment fails the build early.
.gitignore Keeps the repo portable: ignores the large prototype data bundle, __pycache__, notebook checkpoints, and the non-portable local data/ symlinks, while keeping the tracked *.csv split files.

scripts/ (the team's code — deep-dive pages linked)

File What it is
scripts/train.py The team's main artifact: a complete, checkpointed two-stage training loop (KarmaDock ships none). Handles the MDN and RMSD losses, optimizer, early stopping, gradient accumulation, deterministic train/val split, Weights & Biases logging, and --resume.
scripts/run_train.sh Prototype training driver: scratch runs the paper's 2-stage protocol (pipeline P2), finetune continues from the released weights (pipeline P3); it preprocesses the 712-complex proto_train first.
scripts/run_full_train.sh Full-dataset training driver (full_train 23,483 / full_val 2,609). Unlike the prototype driver it reads the curated train/val split via --val_csv / --val_graph_dir rather than a random carve-out.
scripts/run_infer.sh Portable docking driver: preprocess -> dock -> export the three pose variants (uncorrected / FF-relaxed / align-corrected).
scripts/evaluate.sh The separate scoring step: runs the official evaluation.py over every pipeline x pose-variant and writes the RMSD CSVs.
scripts/convert_seminar_to_karmadock.py Re-lays-out the seminar data (<id>_ligand_refined.sdf / <id>_protein_refined.pdb) into the directory shape KarmaDock's preprocessing expects.
scripts/convert_karmadock_to_seminar.py Converts KarmaDock's docked poses into the seminar's results/<ds>/<id>_pred.sdf format, best-pose-first, which is what evaluation.py reads.
scripts/seminar_csv.py Small CSV-schema helper used by the training/preprocess steps to read the seminar and full-split CSV mappings.
scripts/README.md Provenance note: exactly what the team wrote versus the upstream KarmaDock authors' code, stating that no upstream source file was modified.

Authorship & provenance

The key point: public KarmaDock (schrojunzhang/KarmaDock) ships inference code + pretrained weights only — there is NO training script. Our seminar contribution is the training loop, the data adapters, the run wrappers, and the whole HTCondor orchestration built around KarmaDock. We call KarmaDock's modules as-is (model, preprocessing, docking) and did not modify any file inside the upstream KarmaDock/ — it is cloned fresh in the Dockerfile; all of our code lives in scripts/.

.sub, the Docker build, and all analysis/docs. No KarmaDock source file was edited.

Check your understanding

  1. Which parts belong to upstream KarmaDock, and which parts are the seminar team's work?

    Show answer

    Upstream supplies the model, preprocessing and docking utilities, and released weights; the team supplies training, adapters, wrappers, Condor jobs, the Docker build, and analysis documentation.

  2. Why was reducing the requested GPU count a practical fix on the cluster?

    Show answer

    Jobs requesting multiple GPUs remained idle for days, while the prototype could execute on one GPU, so the smaller request made scheduling feasible.

  3. What evidence in the repository helps distinguish authored code from reused upstream code?

    Show answer

    The provenance tables enumerate the scripts and submission files the team created, explain which upstream modules they call, and state that no source inside upstream KarmaDock/ was edited.
