File-by-file guide

This guide covers the meaningful files in the final full-data submission. The (full data) marker distinguishes files specific to the finished full-data work from prototype files and shared infrastructure.

Repository root

File What it is
README.md The final submission report: full-data headline (82.2 % vs released 88.3 %), the three pipelines, results (full-data primary + prototype context), how to reproduce, training parameters, repository layout, and issues.
Dockerfile Builds the reproducible image ahlamloum/karmadock-seminar:v6: miniconda base, clones upstream KarmaDock, unpacks the authors' pre-packed conda env (Zenodo tarball, avoids the conda solver), pip-installs prefetch_generator/rmsd/wandb, copies scripts/, sets PYTHONPATH, and runs a build-time import check.
.gitignore Keeps the repo portable: ignores the prototype data bundle, __pycache__, notebook checkpoints, and non-portable local data/ symlinks; keeps the tracked *.csv split files.

File What it is
scripts/train.py Main artifact: complete checkpointed two-stage training loop (KarmaDock ships none): MDN + RMSD losses, optimizer, early stopping, gradient accumulation, deterministic val split, W&B, --resume.
scripts/train_ddp.py (full data) Multi-GPU (DDP) Stage-2 trainer; full-data Stage 2 was too slow single-GPU, so it was run on 2× A100 (torchrun + NCCL, id-shard, all-reduced metrics, rank-0 checkpointing).
scripts/run_train.sh Prototype training driver: scratch = paper 2-stage (P2), finetune = from released weights (P3); preprocesses proto_train (712).
scripts/run_full_train.sh (full data) Full-dataset training driver (full_train 23,483 / full_val 2,609), reads the curated split via --val_csv/--val_graph_dir.
scripts/run_full_stage2_ddp.sh (full data) Launcher for the DDP Stage-2 run (torchrun wrapper around train_ddp.py).
scripts/run_full_stage2_ddp_v2.sh (full data) Paper-faithful revision of the DDP Stage-2 launcher (weight_decay 0, patience 70); this is the one the final 2× A100 job used.
scripts/run_infer.sh Portable docking driver: preprocess → dock → export the three pose variants (uncorrected / FF / align).
scripts/evaluate.sh Scoring step: runs evaluation.py over every pipeline × pose variant and writes the RMSD CSVs.
scripts/convert_seminar_to_karmadock.py Re-lays seminar data into KarmaDock's expected layout.

Authorship & provenance

The key point: public KarmaDock (schrojunzhang/KarmaDock) ships inference code + pretrained weights only — there is NO training script. Our seminar contribution is the training loop, the data adapters, the run wrappers, and the whole HTCondor orchestration built around KarmaDock. We call KarmaDock's modules as-is (model, preprocessing, docking) and did not modify any file inside the upstream KarmaDock/ — it is cloned fresh in the Dockerfile; all of our code lives in scripts/.

  • Authors': the KarmaDock model + preprocessing/docking utilities + released weights (used as-is).
  • Ours: train.py (there was no trainer), the data adapters, the run wrappers, every condor

.sub, the Docker build, and all analysis/docs. No KarmaDock source file was edited.

Check your understanding

  1. Which parts belong to upstream KarmaDock, and which parts are the seminar team's work?

    Show answer

    Upstream supplies the model, preprocessing and docking utilities, and released weights; the team supplies training, adapters, wrappers, Condor jobs, the Docker build, and analysis documentation.

  2. Why was reducing the requested GPU count a practical fix on the cluster?

    Show answer

    Jobs requesting multiple GPUs remained idle for days, while the prototype could execute on one GPU, so the smaller request made scheduling feasible.

  3. What evidence in the repository helps distinguish authored code from reused upstream code?

    Show answer

    The provenance tables enumerate the scripts and submission files the team created, explain which upstream modules they call, and state that no source inside upstream KarmaDock/ was edited.
