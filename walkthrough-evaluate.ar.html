scripts/evaluate.sh

مسودة ترجمة تقنية أولية للمراجعة. قد تحتاج المصطلحات والصياغة إلى تنقيح.

الغرض

هذا هو مشغّل التقييم الدفعي للنموذج الأولي. يقيّم كل دليل نتائج متاح من حاصل ضرب P1/P2/P3 في الخام/FF/المحاذى باستخدام واجهة مجموعة proto_test نفسها، ثم ينسخ ملف CSV الخام لـP2 إلى اسم النتيجة الرئيس في المستودع.

اختبر فهمك

  1. لماذا يقيّم الغلاف مصفوفة 3×3 بدل استدعاء المقيّم مرة واحدة؟

    اعرض الإجابة

    يطبق مقيّم البيانات نفسه على ثلاثة مسارات نماذج وثلاثة متغيرات وضعيات كي تكون ملفات CSV الخاصة بـRMSD قابلة للمقارنة مباشرة.

  2. لماذا يُنسخ ملف CSV الخام لـP2 إلى proto_test_evaluation.csv؟

    اعرض الإجابة

    هذا الاسم المتعارف عليه هو النتيجة الرئيسة للمستودع، بينما يبقى ملف P2 المصدر ذو الاسم الصريح متاحًا لتتبع المنشأ.

  3. لماذا يغيب توليد الوضعيات عمدًا عن البرنامج؟

    اعرض الإجابة

    التقييم مرحلة مستقلة قابلة لإعادة الإنتاج تستهلك نتائج موجودة ويمكن تكرارها دون إعادة الإرساء المكلف.

موقعه في المنظومة

يستدعيه condor/evaluate.sub بعد مهام الاستدلال الثلاث للنموذج الأولي. ويتوقع evaluation/evaluation.py وproto_test.csv والبنى المرجعية وأشجار الوضعيات تحت results/. ويكتب results/<pipeline>_<variant>_evaluation.csv، ويمكن أن يكتب results/proto_test_evaluation.csv. لا توجد وسائط موضعية ولا متغيرات بيئة مخصصة.

اختبر فهمك

  1. ما الذي يجب أن تنتجه مهام الاستدلال قبل أن يستخدم condor/evaluate.sub هذا الغلاف؟

    اعرض الإجابة

    يجب أن تنشئ أدلة الوضعيات الخام أو FF أو المحاذاة المتاحة لـP1 وP2 وP3 تحت results/.

  2. لماذا يفحص الغلاف مواقع بأسلوب المستودع وأخرى مسطحة؟

    اعرض الإجابة

    قد يحافظ تجهيز HTCondor على أشجار الأدلة أو يضع المدخلات المنقولة عند جذر الصندوق، وكلا التخطيطين صالح.

  3. ما المخرجات التي يمكن أن تغيب من دون إفشال الغلاف كله؟

    اعرض الإجابة

    تُتجاوز ملفات CSV لأدلة الوضعيات المفقودة، كما تُتجاوز النسخة الرئيسة إذا لم توجد نتائج P2 الخام.

شرح الشيفرة

#!/usr/bin/env bash
# evaluate.sh — the scoring step (separate from docking).
# ...
set -euo pipefail

ROOT="$PWD"
EVALPY="$ROOT/evaluation/evaluation.py"

يجعل نمط Bash الصارم المسارات المفقودة وعمليات التقييم الفاشلة توقف المهمة. وبخلاف أغلفة التدريب والاستدلال، لا يستخدم set -x، لذلك لا تظهر إلا الرسائل الصريحة وسجلات المقيّم. يفترض البرنامج أن الدليل الحالي هو المستودع أو جذر المهمة المسطح.

first_existing () {   # <test-flag -f|-d> <candidate>...
    local flag=$1; shift
    local p; for p in "$@"; do [ "$flag" "$p" ] && { echo "$p"; return; }; done
}
CSV=$(first_existing -f "$ROOT/data/proto_test.csv" "$ROOT/proto_test.csv")
REF=$(first_existing -d "$ROOT/data/prototype_model_data/proto_test" "$ROOT/data/proto_test" "$ROOT/proto_test")

[ -n "$CSV" ]    || { echo "ERROR: proto_test.csv not found (looked in data/ and CWD)"; exit 1; }
[ -n "$REF" ]    || { echo "ERROR: proto_test reference dir not found — unzip data/prototype_model_data.zip -d data/prototype_model_data"; exit 1; }
[ -f "$EVALPY" ] || { echo "ERROR: missing $EVALPY"; exit 1; }

قد يحافظ HTCondor على أشجار الأدلة أو يسطّح العناصر المنقولة بحسب سياق التجهيز، لذلك تفحص first_existing عدة مواقع صالحة. وسيطها الأول عامل اختبار، -f للملف أو -d للدليل، وتترك shift المرشحات فقط. تحوّل الحراسات الثلاث نتيجة البحث الفارغة إلى رسالة خطأ قاتلة وواضحة.

eval_one () {   # <poses_dir> <out_csv>
    local poses=$1 out=$2
    local E; E="$(mktemp -d)"
    mkdir -p "$E/data" "$E/results"
    ln -sf "$CSV"   "$E/data/proto_test.csv"
    ln -sf "$REF"   "$E/data/proto_test"
    ln -sf "$poses" "$E/results/proto_test"
    ( cd "$E" && python3 "$EVALPY" --dataset proto_test --no_pb_valid --output_csv "$out" )
    rm -rf "$E"
}

لدى evaluation.py جذور نسبية ثابتة، data/ وresults/، ويشتق المسارات من اسم مجموعة البيانات. بدل نقل البيانات، تبني الدالة عرضًا مؤقتًا بروابط رمزية كي يظهر أي دليل وضعيات باسم results/proto_test. يقصر --no_pb_valid هذه المقارنة الدفعيّة عمدًا على RMSD ويتجنب فحوص PoseBusters المكلفة. ولا يغيّر أمر cd المحاط بقوسين دليل المستدعي.

for tag in p1_baseline p2_scratch p3_finetune; do
  for variant in ":uncorrected" "_ff:ff" "_align:align"; do
    suffix="${variant%%:*}"; label="${variant##*:}"
    poses="$ROOT/results/$tag/proto_test${suffix}"
    if [ ! -d "$poses" ]; then echo "skip $tag/$label (no poses at ${poses#$ROOT/})"; continue; fi
    eval_one "$poses" "$ROOT/results/${tag}_${label}_evaluation.csv"
    echo "wrote results/${tag}_${label}_evaluation.csv"
  done
done

تعرّف الحلقتان المتداخلتان مصفوفة 3×3. يُقسّم كل رمز suffix:label بتوسيع معاملات Bash؛ تزيل %%:* النقطتين وكل ما بعدهما، بينما تزيل ##*: كل شيء حتى آخر نقطتين. تُتجاوز متغيرات الوضعيات المفقودة بدل إفشال المصفوفة كلها.

if [ -f "$ROOT/results/p2_scratch_uncorrected_evaluation.csv" ]; then
  cp "$ROOT/results/p2_scratch_uncorrected_evaluation.csv" "$ROOT/results/proto_test_evaluation.csv"
  echo "wrote results/proto_test_evaluation.csv (= P2 from-scratch, uncorrected)"
fi

تنشئ النسخة النهائية اسم النتيجة الافتراضي المتعارف عليه مع الاحتفاظ بملف CSV المصدر ذي الوسم الصريح.

اختبر فهمك

  1. لماذا تنشئ eval_one دليلًا مؤقتًا من الروابط الرمزية؟

    اعرض الإجابة

    يتوقع مقيّم Python مساري data/proto_test وresults/proto_test النسبيين الثابتين، فتعرض الروابط أي شجرة وضعيات مختارة عبر تلك الواجهة دون نقل البيانات.

  2. كيف ينتج رمز الحلقة suffix:label مسار دليل واسم ملف خرج معًا؟

    اعرض الإجابة

    يستخرج توسيع معاملات Bash اللاحقة قبل النقطتين لـproto_test${suffix} والوسم بعدهما لاسم ملف CSV الخاص بالتقييم.

  3. لماذا يُستخدم --no_pb_valid لهذه المقارنة الدفعيّة؟

    اعرض الإجابة

    يبقي المصفوفة مركزة على RMSD ويتجنب تكرار فحوص PoseBusters المكلفة لكل مسار ومتغير معالجة لاحقة.

محاذير وملاحظات

  • لا تُحذف الأدلة المؤقتة إلا في نهاية eval_one الناجحة؛ ولا يوجد trap، لذلك قد يترك التقييم المنقطع أو الفاشل دليلًا في منطقة النظام المؤقتة.
  • يعطّل هذا الغلاف PoseBusters دائمًا مع أنه مفعّل افتراضيًا في مقيّم Python.
  • يعرض الغلاف البيانات للمقيّم دائمًا باسم proto_test حتى عند المرور بمسارات نماذج مختلفة؛ وهذا صحيح لأن البنى واحدة ولا تتغير إلا التنبؤات.
  • تستبدل إعادة التشغيل ملفات CSV الحالية للتقييم والنسخة الرئيسة.

اختبر فهمك

  1. لماذا قد يترك التقييم الفاشل ملفات مؤقتة؟

    اعرض الإجابة

    لا يحدث التنظيف إلا بعد أمر مقيّم ناجح، ولا يوجد trap يغطي الانقطاع أو الفشل داخل eval_one.

  2. لماذا يكون عرض كل مسار باسم مجموعة البيانات proto_test صحيحًا؟

    اعرض الإجابة

    تبقى البنى المرجعية ومجموعة CSV نفسها؛ ولا تتغير إلا التنبؤات المختارة عبر الرابط الرمزي المؤقت للنتائج.

  3. ماذا يحدث لملفات CSV الحالية عند إعادة تشغيل الغلاف؟

    اعرض الإجابة

    تستبدل التقييمات الناجحة ملفات نتائجها المسماة، ويستبدل ملف P2 الخام النسخة الرئيسة المتعارف عليها.
